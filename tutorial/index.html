Browse Videos

The public landing page of the platform. Any visitor (including non-logged-in users) can see all available videos. This is the first page users see when they arrive on the site.

Search inputTop toolbarFilters the video grid in real-time by title, creator username, or description text.
Sort dropdownTop toolbarSorts cards by: Newest, Cheapest, Most Expensive, or Alphabetical (A–Z).

User Dashboard

User Page

The main control panel for every registered user. It combines content creation tools, profile management, and access control — all on one page. Only accessible when logged in.

SectionAPIFunction
Edit Profile & ContactGET/POST /api/profileCreator fills in: WhatsApp number, bank account (for fiat payouts), EVM wallet address + preferred chain (for X402 crypto payouts), profile bio. This information appears on the creator's public profile page so buyers can verify payment details.
Change PasswordPOST /api/change_passwordUpdates the user's login password. Requires current password for verification. Sends a confirmation email on success. Has a "show/hide" toggle on each field.
My Videos & AccessGET /api/my_videosLists all videos uploaded by this user. Each card shows: title, video ID (copyable), creation date, current price, and the allowlist. Inline "Edit" form lets the creator update title, description, and price without leaving the page.
Upload VideoPOST /api/uploadSubmits an MP4 file. Backend saves it to storage, inserts a DB row, and queues a transcoding job. The video becomes browsable once transcoding finishes (HLS ready).
Grant Access (Allowlist)POST /api/allowCreator manually grants another user permanent access to a video without payment. Useful for reviewers, team members, or giveaways. Has a live username lookup to confirm the user exists before submitting.
The Wallet and Affiliate links are in the navbar, not on this page. Dashboard = content creation tools. Wallet = money. Affiliate = commission system.
  1. User registers → lands on Dashboard → fills in profile (bank/wallet) for payout
  2. Uploads an MP4 → backend transcodes → video appears in Browse
  3. Optionally grants manual access to specific users via Allowlist
  4. Buyers purchase → creator sees earnings in Wallet

Wallet

User Page

The internal ledger-based wallet. Every user has a balance in cents (stored as a BIGINT in the database). No blockchain involved — it's a pure accounting system. All operations are atomic PostgreSQL transactions with row-level locking to prevent double-spending.

SectionAPIFunction
Balance CardGET /api/wallet/balanceShows current balance in USD and IDR equivalent. Also shows the last 3 transaction summaries.
DepositPOST /api/wallet/depositCreates a pending deposit request. The user enters an amount and a note (e.g., bank transfer reference). An admin must approve it for the balance to be credited. Status: pending → approved.
WithdrawPOST /api/wallet/withdrawRequests a withdrawal to the user's registered bank account. Balance is immediately held (deducted). Admin processes the bank transfer then marks it complete. Status: pending → completed (or rejected, which refunds the held amount).
TransferPOST /api/wallet/transferInstant peer-to-peer transfer to another user by username. Both balances update atomically in the same transaction. Rows are locked in alphabetical order to prevent deadlocks.
Pay for VideoPOST /api/wallet/payOne-click purchase of a video using wallet balance. Deducts from buyer, credits creator (at creator_split_bp rate), and optionally pays affiliate commission if a ref_code was in the URL. Grants immediate video access.
Transaction HistoryGET /api/wallet/transactionsFull audit log: type (deposit/withdrawal/transfer_in/transfer_out/purchase/sale/commission), amount, counterparty username, notes, status, timestamp.
Affiliate commission from Wallet Pay: If the buyer arrived via ?ref=username link, the wallet_pay handler also calls commission::process_affiliate_commission() after the purchase commits. The commission is deducted from the creator's wallet and credited to the affiliate.
Transaction status badges: pending (yellow) = awaiting admin action. completed (green) = finalized. rejected (red) = cancelled, balance refunded for withdrawals.

Affiliate Program

User Page

Three-tab interface combining the creator-side configuration and the affiliate-side earnings dashboard. Any user can be both a creator (who pays commissions) and an affiliate (who earns commissions) simultaneously.

TabWho Uses ItFunction
My EarningsAffiliate (User B)Shows total lifetime commission earned. Table of every commission event: video purchased, buyer username, amount, payment method (wallet/x402/fiat), and timestamp. Commission is already in the wallet balance — this is just the history.
My VideosCreator (User A)Configure the affiliate program per video. Select a video, set commission percentage (0–90%), toggle enabled/disabled, and save. The creator can set different rates for different videos or disable the program on specific titles.
Get LinksAffiliate (User B)Generate a referral link for any video with an active affiliate program. The link format is: /public/watch.html?video_id=X&ref=USERNAME. Includes a copy button. Also shows whether the program is currently active for that video.
  1. Creator (Alice) goes to "My Videos" tab → selects her video → sets 15% commission → enables
  2. Affiliate (Bob) goes to "Get Links" tab → selects Alice's video → copies referral link
  3. Bob shares the link on social media
  4. Buyer (Carol) clicks the link → sees "Referred by @bob" notice → purchases
  5. Commission is automatically deducted from Alice's wallet and credited to Bob's wallet
  6. Bob sees the commission in "My Earnings" tab and in his Wallet transaction history

Watch / Payment Page

User Page

The video detail and purchase page. Opened when a user clicks a video card on the Browse page. This is where all three payment paths converge.

StateShown WhenWhat User Sees
Video InfoAlwaysTitle, description, creator username, price, creator contact info (WhatsApp, bank, wallet). Creator's public profile description.
HLS PlayerAfter access grantedNative HTML5 video player with HLS.js. Press F for fullscreen. Watermark overlaid with buyer's username.
Payment PanelNo access yet3-tab payment interface: (1) Wallet Pay — one-click if balance sufficient, (2) Crypto (X402) — MetaMask/EVM wallet, (3) Fiat — Stripe/PayPal/Midtrans/Xendit checkout. All tabs respect the ?ref= referral code.
Referral NoticeWhen ?ref= in URLBlue info badge: "Referred by @username. They earn a commission when you purchase." Shown inside the payment panel.
Free VideoPrice = 0Playback granted immediately without payment. No payment panel shown.
The ?ref= parameter is captured once on page load into REF_CODE and passed to all three payment methods automatically. The buyer never needs to do anything special — the referral is tracked transparently.

Login & Register

User Page

Standard auth pages. All passwords are hashed with Argon2 in PostgreSQL. Sessions are HMAC-SHA256 signed cookies.

PageRouteFunction
Register/auth/registerCreates a new user account. Username must be unique. Password minimum 8 characters.
Login/auth/loginAuthenticates user, creates a signed session cookie (TTL configurable via SESSION_TOKEN_TTL env var).
Forgot Password/auth/forgotSends a password-reset link to the user's email address. Requires SMTP to be configured.
Reset Password/auth/resetConsumes the one-time reset token from the email link. Sets a new password.

Payment Monitoring (Pembayaran)

Admin Page

The fiat payment monitoring and creator payout management page. Shows all invoices created by Stripe, PayPal, Midtrans, and Xendit providers. Admin can track payment status and trigger creator payouts (disburse).

UI ElementFunctionDetail
Status badges (totals)OverviewShows total count of All / Paid / Pending / Failed invoices at the top of the page.
Provider filterFilterShow only invoices from a specific provider: Stripe, PayPal, Midtrans, Xendit.
Status filterFilterFilter by: pending / paid / failed / expired / cancelled.
Limit selectorFilterShow 50 / 100 / 500 rows.
Invoice tableMain dataColumns: Invoice UID (truncated), Provider (color-coded badge), Buyer (username + email), Video title, Amount, Status, Created date, Paid date, Disburse status, Action button.
Disburse buttonActionAppears only on paid-but-not-yet-disbursed invoices. Opens a confirmation modal. Xendit: triggers automatic bank transfer via Xendit API. Other providers: marks as "manual" (admin transfers themselves through the provider's dashboard).
Disburse modalConfirmationShows invoice details, buyer, video, amount, and creator bank account. For Xendit with bank account set: shows "auto-disburse" message. For Xendit without bank: warns cannot auto-disburse. For others: shows "manual transfer" reminder.
Xendit is the only provider with automatic bank disbursement. For Stripe/PayPal/Midtrans, the admin manually transfers funds through those providers' own dashboards, then clicks "Disburse" here to mark it as done in the system.

Settings — SMTP & Admin Password

Admin Page

Two forms on one page: email server configuration and admin password change. Both are admin-only.

SectionAPIFunction
Enable email togglePOST /admin/smtpIf off, emails are only logged to server console — nothing is actually sent. Safe for development environments.
SMTP configurationPOST /admin/smtpSets: SMTP Host, Port, Username, Password (App Password for Gmail), From Email, From Name, TLS toggle. Saved to the smtp_settings table in the database (not to .env).
Test EmailPOST /admin/smtpSends a real test email to a specified address using the current configuration. The settings are saved first, then the test is sent. Good for verifying SMTP credentials.
Ganti Password AdminPOST /admin/change_passwordChanges the admin login password. Requires the current password. Updates users.password_hash in the database (NOT the .env file). Sends email notification on success.
Panduan Konfigurasi—Read-only reference guide showing correct host/port settings for Gmail, Mailgun, Mailtrap, and Brevo/Sendinblue.
The .env file is never modified by the application. All runtime settings (SMTP, password) are stored in the database. The .env file only sets the initial bootstrap values.

Admin Wallet Management

Admin Page

Admin view of the internal wallet system. Admins approve user deposit requests, confirm withdrawals, and reject transactions. This page processes the manual money movement side of the wallet system.

SectionAPIFunction
Stats row—Quick counts: Total Pending Deposits, Total Pending Withdrawals, Total amount waiting for action (in USD).
Filter barGET /admin/wallet/transactionsFilter by: transaction type (deposit/withdrawal/transfer/etc.), status (pending/completed/rejected), limit.
Transaction table—All wallet transactions across all users. Columns: user, type, amount, notes (bank transfer reference, etc.), status badge, date, action buttons.
Approve buttonPOST /admin/wallet/transactions/:id/approveApproves a pending deposit. Credits the amount to the user's balance. Status: pending → completed. Shows a modal to optionally add an admin note.
Complete buttonPOST /admin/wallet/transactions/:id/completeMarks a withdrawal as paid (the admin has already sent the bank transfer). Balance was already held when the withdrawal was created. Status: pending → completed.
Reject buttonPOST /admin/wallet/transactions/:id/rejectRejects a pending transaction. For withdrawals, refunds the held balance back to the user. Status: pending → rejected.
  1. User goes to Wallet page → clicks Deposit → enters amount and bank ref number → submits
  2. Admin sees pending deposit in Admin Wallet → verifies the user's bank transfer was received
  3. Admin clicks Approve → user's balance is credited
  4. User can now spend the balance on videos or transfer to others
  5. User requests withdrawal → balance held immediately
  6. Admin processes bank transfer → clicks Complete in Admin Wallet
